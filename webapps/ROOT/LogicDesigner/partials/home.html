Architecture

A quick overview of the key components:

Espresso simplifies and accelerates creating RESTful servers, with declarative definition of the services typically found in an App Server:

  1. Security: controls end-point access, and row/column security.

  2. API: point-and-click document-oriented endpoints, complementing the default API

  3. Logic: database integrity with a combination of spreadsheet-like Rules and server-side JavaScript.

  4. Integration: resources can combine data from multiple sources (SQL, Mongo and RESTful), and coordinate updates between them.

Specify your settings in the Designer; activation is instantaneous - no code gen, no deployment.

Use the Live Browser, created from the schema, to test your API, and for back-office database maintenance.

Flexible Deployment - both the RESTful service and database can be in the cloud, or on-premise (VMware image).

Use your own existing database, or the pre-supplied empty database in the Your Database project.

Key Do's and Don'ts

You will probably pick up API Creation quite quickly, since it's a point and click technique for building a commonly understood API. Espresso's most unique feature is Reactive Programming for business logic, so it's important to be aware of that.

Don't

Do
Code Logic in client Specify Logic in Server to:
  • Centralizes your logic for re-use, including for non-UI clients such as other systems.
  • Improves performance by reducing the amount of data which must be sent to the client
Code Logic in Events Think Spreadsheet - Spreadsheet-like rules are orders of magnitude more expressive than events

Event-driven programming is a common programming model, and fully supported so there are no boundaries on what you can address.

But it's a big mistake to use it instead of Reactive Programming spreadsheet-like rules. Reactive automates Change Detection / Propagation, which you do manually in event handlers.

For example, it is a common error to do sum/count rollups in event handlers - sum/count rules are much easier, and are apt to perform better.







Espresso Logic enables you to rapidly deploy a REST/JSON service, using

  1. Business Rules for integration, logic and security - meaningful reductions in tedious and repetitive code

  2. Server-side JavaScript for integrating enterprise data, complex logic - whatever is required


Espresso Logic is a service, so there's nothing to install or configure. It consists of two main components:

  1. Use the Espresso Designer (you're in it now) to manage your Project
    1. A project corresponds to a RESTful service:
      1. It is the grouping of your database location, and your API, security, integration and rule settings.
      2. Your account can include many projects
      3. They are all running, but you manage the contents of only the current project at one time, selected via the combo box at the top
    2. Create a new project (combo box at top) and connect to your database. This creates a default API and UI
    3. Then use the Navigation Bar to the left (or click the diagram above) to specify your project settings (API, security, integration and rules)


  2. The Espresso Server (big box above)
    1. stores these settings (you can import / export Projects as JSON)
    2. and enforces them in the course of processing REST retrieval / update requests



Live Browser
The Live Browser is a multi-table HTML5 user interface:

  1. It is automatically created from your schema when you connect, and is responsive to schema changes

  2. It enforces your logic and security since it uses your RESTful API.


Live API
When you connect, the system creates Default End-points for each table, view and stored procedure. You can control access using Security.

You can also define Custom End-points, for example for custom apps, or for data integration Web Services. We have pre-created samples so you can explore them.

Observe:

  1. Custom Resources provide a tree-like document model

  2. You can project / alias (rename) the objects and attributes, so your API is truly custom, not merely a flat reflection of your relational tables

  3. Logic and Security apply "underneath"" the RESTful API, so are enforced for Live Browser, or any RESTful API call, whether through default or Custom End-points

  4. Test them in the REST Lab


Live Logic



  1. Logic enforces database integrity on RESTful update requests - business rules for derivations, validations, integration, and events.

  2. Reactive Logic is conceptually similar to spreadsheet cell formulas:
    • You bind expressions to database columns, e.g.
      Derive balance as sum(purchaseorderList.amount_total where paid=false)
    • Invocation is automatic - the engine (not your code) calls the logic - it never forgets
    • Automatic Change Propagation to dependent data (e.g., the balance is reduced when the order is paid)
    • Automated SQL and transaction bracketing for Change Propagation to related tables

    Reactive eliminates tedious code for Change Detection and Propagation, Persistence and Transaction Management. As a result, a small handful of reactive logic is equivalent to many hundreds of lines of conventional code.

  3. Reactive works in conjunction with server side JavaScript, based on an automatically created Object Model with attribute/object accessors, persistence, and events for insert, update and delete

    • Extensible Rule Types can be built using your own JavaScript libraries (via Project settings).


  4. Check Credit is a typical example: it
    • Adjusts the Customer Balance when Order Details are changed
    • Validates your data, with roll back and an error message (e.g., Credit Limit exceeded)
    • More complex examples here

Integration
Espresso provides a set of integration services to help you manage distributed information:

  1. Mashup - You can define Resources that combine data from multiple Data Sources, whether SQL, RESTful or other.

  2. Business Object Synchronization - You can use Business Logic to synchronize sets of related rows (Business Objects) between Data Sources.
      ​
    1. You can post data to another server, for example regulatory compliance, or to post changes to an Enterprise Service Bus
    2. You can acquire data from another source, both to use in processing a transaction, or to store locally (with rule enforcement)
Synchronization services leverage Resources, where you can project / alias data to transform data to match external formats. Learn more about integration here.


Update Operation

This diagram illustrates how the Espresso Logic executes your business logic in response to REST post/put/delete requests (see here for more information). The update life cycle occurs at both the request and row level:

  1. The Request Life Cycle receives the request, marshals the JSON data into row objects, and initiates row logic processing.  It also provides 2 events, each called once per request.

  2. The Row Logic is called once for each marshaled row, and for logic chaining (e.g., sum/count adjustments).  This runs your logic, and raises various events as shown.

  3. The Row Commit Logic is a final pass through each row, invoking Commit Validations and Commit Events.
Row Events are passed row, oldRow and logicContext.
